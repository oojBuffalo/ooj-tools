# dox — deterministic AGENTS.md tree engine

A hierarchy of `AGENTS.md` files acts as a **map** of a codebase, so a coding agent
loads only the docs relevant to the path it's touching instead of the whole repo.
This keeps the agent's context small, stops it duplicating code or breaking
conventions, and onboards it to each area in a few hundred tokens.

The original [DOX](https://github.com/agent0ai/dox) is a single `AGENTS.md` that
*asks the model* to maintain this map by hand. That's a prompt — the model can
skip reading or updating the docs and nothing stops it. **This is the mechanistic
version.** The structure and index are derived from the filesystem, drift is
caught by a script, and the model is only ever asked to write prose — and only
when the tool says it must.

## The key idea: split the three jobs

A single AGENTS.md is really doing three jobs. Only one needs a model.

| Job | Needs a model? | Who does it here |
|-----|----------------|------------------|
| **J1** load the right docs for a path | no | `dox context` (path → ancestor chain) |
| **J2** keep the structural index accurate | no | `dox sync` (filesystem → index block) |
| **J3** keep the prose accurate | yes | the agent, only when `dox check` flags it |

J1 and J2 are pure functions of the path and the filesystem, so they're scripted
and deterministic. J3 is the only generative step, and it's gated behind a
deterministic check.

## Install / build

The engine is a zero-dependency TypeScript package that runs on Node - the
language the agent harnesses themselves are built in, so it's already present
wherever a coding agent runs (and `npx`-installable, like the original DOX).

```
npm install        # dev deps (typescript only)
npm run build      # compile src/ -> dist/
node dist/cli.js   # or, once published / linked:  npx dox
```

## CLI

```
dox init       scaffold .dox.json + a root AGENTS.md, then sync
dox sync       regenerate the managed index block in every AGENTS.md
dox check      report drift; exit 1 on hard issues (--strict fails on warnings too)
dox context P  print the root→P AGENTS.md chain (what the agent should read)
```

Drift `check` reports:
- `MISSING`  — a directory has code but no `AGENTS.md`
- `STALE`    — an index block is out of date (auto-fixed by `sync`)
- `DANGLING` — an `AGENTS.md` links to a path that no longer exists
- `PROSE?`   — code in a folder changed but its `AGENTS.md` didn't *(warning; `--strict` to enforce)*

## What goes in each file (DOX video conventions)

Every `AGENTS.md` documents **one domain (folder)** with these canonical sections,
then ends with a machine-generated **Child Docs Index**:

```markdown
# payments

## Purpose
Stripe + ledger. All money math goes through `money.py` (integer cents only).

## Ownership
Parent: services/. Owns charge + refund flows.

## Local contracts (rules)
- All amounts are integer cents. Never floats.

## Work guidance
- New providers go under `providers/`, one file each.

## Test & verify
- `pytest payments/ -q` must pass; run the ledger reconciliation script.

<!-- dox:index -->        ← regenerated by `dox sync`, never hand-edit between markers
## Child Docs Index
- [`webhooks/`](webhooks/AGENTS.md) - inbound Stripe events; idempotent by event id
<!-- /dox:index -->
```

`dox check` enforces the section structure (reports `SECTIONS` when one is
missing). Each child's **Purpose** line is pulled up into the parent's index
automatically — **write a folder's purpose once, it propagates upward.**

### Per-file documentation (per-folder opt-in)
A folder that wants a doc per source file (the video's `screen.py.md` pattern)
adds the marker `<!-- dox:per-file -->` to its `AGENTS.md`. Then `dox check`
requires a sibling `<file>.md` for every source file (`FILEDOC` on a miss) and
`dox sync` lists them in that folder's Child Docs Index. The rule lives in that
one `AGENTS.md`, so it applies only there.

## Enforcement layers (pick any; they all call the same CLI)

The CLI is the single source of truth. Each environment wires it in with its own
deterministic mechanism.

### Git (agent-agnostic — the portable layer)
Runs for *any* committer: Claude Code, Codex, Cursor, or a human.
```bash
cp dox/hooks/git/pre-commit .git/hooks/pre-commit   # or: git config core.hooksPath dox/hooks/git
```
On commit it runs `dox sync` (auto-fixes + restages the index) then `dox check`,
and blocks the commit on hard drift.

### Claude Code (in-loop — the most effective layer)
dox ships as a Claude Code plugin (`.claude-plugin/plugin.json` + `hooks/hooks.json`).
Install it and both hooks register automatically — no settings to hand-edit:
- **PreToolUse** on `Edit|Write|MultiEdit` → injects the `dox context` chain for the
  edited path, so the relevant contracts are *always* in context (J1, mechanised).
- **Stop** → auto-`sync`, then blocks finishing on hard drift (J3 gate).

The hooks resolve the engine from `${CLAUDE_PLUGIN_ROOT}` and operate on your repo via
`dox -C "$CLAUDE_PROJECT_DIR"`. For a vendored (non-plugin) install, merge
`dox/hooks/claude/settings.snippet.json` into `.claude/settings.json` instead.

### CI
Run `dox check` as a job — a backstop if a hook was bypassed.

## Why this beats a prompt
- **Determinism where it's possible.** Loading docs and maintaining the index never
  depend on model judgement.
- **The model is scoped to prose,** and only when a script demands it.
- **It holds for any agent** — the git layer enforces at the VCS boundary, below the
  agent entirely.

## Extending to other harnesses
The CLI is the integration point; adding a harness means wiring `dox context`
(read-side) and `dox check` (gate) into that harness's hook system. Candidates to
research: Codex CLI hooks, Cursor rules/`.cursor`, OpenCode, Aider's lint/commit
hooks. The universal fallback is always the git layer above.

Zero runtime dependencies — TypeScript compiled to ESM, run on the Node the
agent harness already provides.
